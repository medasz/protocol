# ICMP 架构现状分析

## 1. 原始代码问题概览

重构前，代码主要集中在 `icmp_m.go` 和 `icmp_s.go` 两个文件内，存在以下共性问题：

- 单文件承担参数解析、协议解析、报文构造、网卡选择、系统命令执行、日志输出等多种职责。
- `main` 包直接依赖 `pcap`、`gopacket`、`cmd.exe` 和系统路由/ARP 命令，导致核心逻辑无法独立测试。
- ICMP 协议字段、以太网/IP/ICMP 构包逻辑散落在业务流程中，没有统一抽象。
- 系统资源访问方式固定，缺乏依赖注入点，导致网络收发和系统解析几乎无法模拟测试。
- `master` / `slave` 的流程控制与 IO 细节耦合，后续扩展新的执行器、传输介质或消息格式成本较高。

## 2. 原始模块划分

原始结构可简化为：

- `main` 层：CLI、业务调度、协议解析、网络收发、终端输入输出
- `slave` 侧：额外包含 `cmd.exe` 管道控制、默认网关解析、ARP 查询

这会导致模块边界模糊，几乎所有改动都需要触碰顶层入口文件。

## 3. 依赖关系问题

原始依赖呈现“入口直连一切”的扁平结构：

```text
main
 ├─ flag / os / fmt
 ├─ gopacket / pcap
 ├─ net
 ├─ os/exec
 └─ 协议构造 + 业务流程 + 系统资源访问
```

主要问题：

- 没有稳定的抽象边界，协议层无法脱离 `pcap` 生存。
- 依赖注入缺失，测试只能走真实系统环境。
- 传输层与应用层双向“知识泄漏”，应用层知道太多网络细节。

## 4. 代码冗余点

- Echo Request / Echo Reply 的构造逻辑与字段拷贝操作重复出现。
- 源 IP、设备、网关、ARP 等解析逻辑散落在业务流程中，不利于复用。
- 输出复制、payload 拷贝、防御性校验在不同位置重复实现。

## 5. 耦合问题

- `master` 的命令读取直接绑死标准输入。
- `slave` 的命令执行直接绑死 `cmd.exe`。
- `transport` 细节直接嵌在 `runMaster()` / `runSlave()` 中。
- 错误处理策略不统一，既有直接返回，也有过去遗留的 panic 风格。

## 6. 性能与可维护性瓶颈

- 每次发送和接收的协议解析/构包逻辑重复拼装，难以集中优化。
- 网络读取、系统命令解析和业务处理混在一个调用链里，定位性能热点困难。
- 缺少小粒度单元测试，回归验证主要依赖人工联调，迭代成本高。

## 7. 重构目标

- 分离协议层、传输层、应用层、系统适配层和 CLI 层。
- 所有核心逻辑都通过接口连接，避免跨层直接依赖具体实现。
- 为 `pcap`、标准输入输出、`cmd.exe`、系统命令解析提供可替换接缝。
- 将可测试的纯逻辑尽可能下沉，提升单元测试覆盖率和回归效率。
